# 系统能力升级方案

## 1. 升级目标

当前项目已经完成校园零食平台 MVP 底座。本次升级目标是把系统从“能演示”推进到“可持续二开、可试运营、可扩展”的工程结构。

核心方向：

```text
学生快速下单
代理点高效履约
后台清晰运营
多校区低成本复制
```

## 2. 后端分层方案

推荐结构：

```text
routes        接口入口
services      业务编排
repositories  数据读写
constants     状态和枚举
utils         通用响应、校验、错误
middlewares   鉴权、日志、异常处理
```

本次优先落地：

- constants
- utils
- services
- 订单状态机
- 状态日志迁移脚本

## 3. 订单状态机

订单不能随意改状态，必须按照状态机流转。

状态设计：

```text
pending_confirm   待确认
pending_accept    待接单
accepted          已接单
preparing         备货中
delivering        配送中
waiting_pickup    待自提
completed         已完成
cancelled         已取消
after_sale        售后处理中
closed            已关闭
```

允许流转：

```text
pending_confirm -> pending_accept
pending_confirm -> cancelled
pending_accept -> accepted
pending_accept -> cancelled
accepted -> preparing
accepted -> delivering
accepted -> waiting_pickup
preparing -> delivering
preparing -> waiting_pickup
delivering -> completed
waiting_pickup -> completed
completed -> after_sale
after_sale -> closed
cancelled -> closed
```

## 4. 代理点履约模型

后续建议将代理点分单从手动选择升级为规则分配：

```text

→ 判断代理点营业状态
→ 判断代理点商品数量
→ 分配订单
→ 代理点接单
→ 超时提醒或转派
```

建议后续新增：

```text
agent_building_scope
agent_work_status
agent_product_count
agent_dispatch_log
```
